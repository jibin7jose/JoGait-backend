# Step 4: Sync Protocol & Idempotency

This guide describes how the backend handles offline-first data uploads reliably without record duplication.

---

## 1. Idempotency Check (Avoiding Duplicates)

When connection drops during a sync session, the mobile app retries the upload. The server uses a unique `sessionId` (UUID generated by the client) to guarantee idempotency.

```typescript
// Pseudo-code in controller:
const existingSession = await prisma.session.findUnique({
  where: { id: sessionId }
});

if (existingSession) {
  // Return success without duplicating records
  return res.status(200).json({ status: 'already_synced', sessionId });
}
```

## 2. Transaction Integrity

A session upload contains both metadata (score, start time) and a time-series list of metrics. To ensure data is not partially written:

```typescript
await prisma.$transaction([
  prisma.session.create({
    data: { id: sessionId, patientId, deviceId, startTime, endTime, scoreSummary }
  }),
  prisma.metric.createMany({
    data: metrics.map((m: any) => ({
      sessionId,
      metricType: m.metricType,
      value: m.value,
      unit: m.unit,
      timestamp: new Date(m.timestamp)
    }))
  })
]);
```

## 3. Rate Limiting and Data Compression

*   **Optimization**: Ensure the mobile app compresses raw metrics before uploading.
*   **Rate Limits**: Configure middleware (`express-rate-limit`) to limit patients to 20 uploads per hour, protecting database load.
